Section 01

Executive Summary

DB Agent is a local-first GRC (Governance, Risk & Compliance) automation platform built for compliance consultants managing multiple clients. This document covers everything implemented — what was built, how it works, and the reasoning behind each decision.

Core objective

Turn a compliance consultant from someone who writes and edits compliance programs into someone who reviews and approves them — so they can serve more clients, faster, with better output quality than any manual process.

Section 02

What DB Agent Does — End to End

A complete GRC program for a client, from the first onboarding question to a packaged audit-ready output.

Phase 0 — Framework Selection & Onboarding
User fills 20 onboarding fields (company name, industry, cloud providers, data types, MFA status, access model, etc.). The system derives top risks from these inputs and unlocks downstream phases.
Phase 1 — Policy Generation
Multi-stage pipeline generates 25 client-specific policies. Previously: template + regex. Now: 6-agent AI chain — orchestrator, writer, critic, rewriter, with QA cross-check at the end.
Phase 2 — Risk Assessment
Risk Analyst agent reads the approved policies and generates a risk register where every risk cites the actual policy that controls it — or flags the gap where no policy provides coverage.
Phase 3 — Vendor Assessment
Vendor Analyst agent reads both the risk register and the policies. Every vendor assessment references specific risk IDs. Assessment questions target the actual control gaps found in the risk register.
Phase 4 — Control Mapping
Framework controls (SOC 2, ISO 27001) are mapped to policies, risks, and vendors. Every control is traceable back to a specific policy section and risk record.
Phase 5 — Evidence Tracker
Consultant uploads evidence for each compliance task. Adding or removing evidence automatically triggers audit QA re-scoring and targeted re-enhancement of affected risks and vendors via the dependency graph.
Phase 6 — Output & Export
Full compliance program packaged: policy pack, risk register, vendor assessments, control mapping, audit readiness report.
Phase 7 — Intelligence Centre
Quality monitoring dashboard. Shows scores per section, runs targeted AI improvement passes on demand, and displays what changed and why.

Section 03

The Problem with Single-Call AI

Before the multi-agent system, the AI pipeline made three isolated Claude API calls — one for policies, one for risks, one for vendors. They did not share context or communicate.

The core problem: no shared context

Each agent received only the raw onboarding fields. None of them knew what the others had written. The result was a set of documents that existed independently of each other — not a connected compliance program.

What "Isolated" Looked Like in Practice

Before — Policy (isolated)

"GreenHat Assurance must implement access controls appropriate to the risk level of the data handled. The organization should ensure MFA is enforced where possible."

After — Policy (connected)

"GreenHat Assurance enforces MFA for all workforce access via Okta SSO across AWS and GCP. Service accounts are excluded — this is documented as a control gap (RISK-001) requiring remediation before audit."

Before — Risk (isolated, didn't read policy)

"Unauthorized account access — GreenHat's identity controls may have gaps that increase exposure to credential-based attacks."

After — Risk (reads actual policy)

"RISK-001 — Unauthorized access. GreenHat's POL-003 mandates MFA via Okta but service accounts are explicitly excluded in Section 3.2. This is an open control gap."

Before — Vendor (isolated)

"Okta — standard IAM risks apply. Assess MFA coverage and administrative access controls."

After — Vendor (reads risk register)

"Okta directly contributes to RISK-001 (service account MFA gap). Question 4: Does Okta enforce MFA for service accounts in GreenHat's AWS and GCP tenants?"

Section 05

Agent-by-Agent Breakdown

1
Orchestrator — New-CompanyBrief
Reads all onboarding data. Produces a structured company brief shared by all downstream agents.
Output fields company name, industry, full tech stack (every tool named), data types, frameworks, headcount band, scope, implied key risks, documentation tone
2
Policy Writer — Invoke-PolicyWriterAgent
Receives the company brief. Rewrites each policy body and executive summary in batches of 5.
Key prompt rules Name the company in every section. Name every tool from the tech stack. Subsection headings on their own line. 3+ must-statements become numbered lists. Every control must be auditable.
3
Policy Critic — Invoke-PolicyCriticAgent
Scores every policy 0–100. Returns specific flags for each issue found.
Checks for Generic terms ("the organization"), missing tool names, inline headings ("1.1 Heading. Content..."), vague non-auditable control language. Policies scoring below 75 go to the rewriter.
4
Policy Rewriter — Invoke-PolicyRewriterAgent
Fixes only the policies that failed the critic. Surgical — does not touch passing policies.
Efficiency On a typical run, 3–8 policies fail the critic. The rewriter makes 3–8 targeted API calls rather than rewriting all 25. If all policies pass, this stage is skipped entirely.
5
Risk Analyst — Invoke-RiskAnalystAgent
Reads the final approved policy text. Writes risks that cite real policy IDs and section content.
What changes vs before Previously: risks were written with only raw onboarding context. Now: the agent has the actual policy documents and must cite which policy provides control coverage — or explicitly flag that no policy covers the risk.
6
Vendor Analyst — Invoke-VendorAnalystAgent
Reads both the risk register and the policies. Writes vendor assessments that reference real risk IDs.
What changes vs before Previously: vendor questions were generic per service type. Now: questions target the actual control gaps identified in the risk register for each vendor's specific service.
QA
QA Cross-Check — Invoke-QaCrossCheckAgent
Reads all outputs together. Checks that the program is internally consistent.
Checks Risks referencing non-existent policy IDs. Vendors with no risk references. Remaining generic language. Missing company name in key fields. Returns a pass/fail with specific issues.

Section 06

Dependency-Aware Auto-Updates

Instead of reprocessing everything when data changes, the system resolves exactly which records are affected and updates only those — using a live dependency graph built from the current client data.

The Dependency Chain

Evidence item
  → task_id (e.g. "T-001")
    → task.soc2_criteria (e.g. "CC6.8")
      → controls where framework_mapping contains "CC6.8"
        → control.linked_policies → specific policy IDs
          → risks where linked_policies contains those IDs
          → vendors where linked_controls contains those control IDs

Three Trigger Types

Change typeResolution methodWhat updates
Evidence added / removed task_id → criteria → controls → policies → risk IDs + vendor IDs Only risks and vendors linked to those policies
Onboarding field changed CH_FIELD_RISK_CATEGORY map → risk category filter → risk IDs Only risks in the affected category
Vendor record changed Vendor ID directly from change event Only that specific vendor

Selective Regeneration API Endpoints

EndpointScopeAI enhanced
POST /api/clients/:id/process-targetedSpecific policy IDs + risk IDs + vendor IDsYes, per-record
POST /api/clients/:id/process-risksFull risk sectionYes (Risk Analyst Agent)
POST /api/clients/:id/process-vendorsFull vendor sectionYes (Vendor Analyst Agent)
POST /api/clients/:id/process-auditAudit QA onlyNo (instant rule-based)
POST /api/clients/:id/process-policy/:idSingle policy AI rewriteYes
4-second debounce + merge

If a user edits 3 onboarding fields in quick succession, all three impacts are merged into one API call that fires 4 seconds after the last change. A duplicate call for the same section is blocked if one is already in flight.

Before vs After — Output Quality

Policy Body — Access Control

Before (template + regex)

"GreenHat Assurance must implement access controls appropriate to the risk level of the data handled. The organization should ensure MFA is enforced where possible. Access must be reviewed periodically."

After (Policy Writer Agent)

"GreenHat Assurance enforces MFA for all workforce access via Okta SSO across AWS and GCP. Service accounts are a documented gap (RISK-001). The IT Security lead must enforce Okta MFA policies on service accounts by Q2 2026. Access reviews run quarterly via Okta access reports."

Risk Record — Identity & Access

Before (AI, no policy context)

"Unauthorized account access — GreenHat's identity controls may have gaps. Treatment: enforce MFA, review access controls regularly."

After (Risk Analyst reads policies)

"RISK-001 — Unauthorized access. POL-003 (Access Control) mandates Okta MFA but Section 3.2 explicitly excludes service accounts. Existing control: POL-003. Gap: service account coverage. Treatment: 1. Update POL-003 Section 3.2. 2. Enable Okta MFA for all service accounts in AWS IAM."

Vendor Assessment — Okta

Before (AI, no risk context)

"Okta — identity provider. Assess MFA coverage and administrative access. Standard IAM vendor risks apply."

After (Vendor Analyst reads risk register)

"Okta directly contributes to RISK-001 (service account MFA gap identified in POL-003). Q4: Does Okta enforce MFA for service accounts in GreenHat's AWS and GCP tenants? Q5: Can Okta generate audit logs showing MFA enforcement per account type?"

Section 10

Roadmap — What's Next

Two capability gaps remain before DB Agent is a full industry-standard GRC platform. The quality gap is now closed. The automation gap requires integrations.

Priority 1 — Automated Evidence Collection (Integration Layer)

The single biggest gap vs Vanta/Drata

Connect to AWS, GitHub, Okta, Google Workspace APIs and pull evidence automatically. Instead of uploading a screenshot proving MFA is on, the system checks the live API and passes or fails automatically. This is a 3–6 month build per integration.

IntegrationEvidence it providesPriority
GitHubMFA enforcement, branch protection, access reviewsHigh
Okta / Google WorkspaceMFA status, user provisioning, access logsHigh
AWS / GCPEncryption, logging, IAM policies, S3 configHigh
SlackSecurity awareness comms, incident notificationsMedium
Jira / LinearVulnerability tracking, remediation evidenceMedium

Priority 2 — Multi-User & Audit Room

Allow clients to log in and fill their own onboarding. Allow auditors to request evidence directly in the platform. Role-based access: consultant, client admin, auditor (read-only).

Priority 3 — Critic-Rewriter Loop Enhancement

Currently the critic runs once. A future version runs the critic → rewriter → critic loop up to 3 times until all policies score above 85. This produces measurably higher quality output at the cost of 2–3 additional API calls per failing policy.

Priority 4 — Continuous Monitoring

Daily scheduled checks that controls are still passing (once integrations exist). Risk scores update automatically when new evidence arrives or when a control check fails. Consultant gets notified of regressions.

Current position

DB Agent has closed the quality gap against Vanta and Drata — no GRC platform on the market generates AI-written, internally-consistent, client-specific compliance documentation with agent-to-agent context sharing. The remaining gap is on the automation side — the integration layer that auto-collects evidence. That is the next build phase.
